# ExpLine 需求实现对照

## 总体结论

当前实现已经覆盖 ExpLine 的核心记录闭环：

```text
包装运行实验 -> 捕获 Git 变化 -> 读取项目说明和父实验 -> AI 生成语义记录 -> 保存实验节点 -> 更新索引
```

也就是说，项目已经具备“自动生成可读实验语义记录”的基础能力。

但需求文档中较大的后半部分仍未完成，主要集中在：

1. 静态 HTML 可视化页面。
2. `rebuild` 重建能力。
3. `list`、`site` 等命令。
4. Git 同步和多分支 merge 后的恢复流程。
5. 可视化页面中的节点点击、右键复制、搜索和链路高亮。

另外，需求文档文件名是 `v0.3`，但文档标题仍写着 `ExpLine 需求文档 v0.2`，需要后续统一。

## 已实现

1. `expline init` 可以创建 `.expline/`、配置文件、Prompt 模板、项目摘要和索引文件。
2. `expline init --no-ai` 和 `expline rescan --no-ai` 可以跳过 AI，使用本地回退摘要器。
3. `expline rescan` 可以重新生成项目语义说明。
4. `expline run -- <command>` 可以包装并执行实验命令。
5. 每次 `run` 会生成一个新的实验目录和实验记录。
6. 实验记录包含 `command.txt`、`diff.patch`、`changed_files.txt`、`record.json`、`record.ai.md`、`record.md`。
7. 实验记录会保存 Git commit、branch、dirty 状态、diff 和变更文件列表。
8. 支持非 Git 项目降级运行。
9. 支持默认父实验推断：当前 commit 或祖先 commit 上最近实验。
10. 支持 `--parent EXP-xxxx` 手动指定父实验。
11. 指定或推断出的父实验不存在时，会在执行用户命令前报错，避免生成孤立链路。
12. 支持跨分支父实验比较：指定父实验时，会比较父实验 commit 到当前 commit 的 diff。
13. 支持 AI 生成结构化实验报告，失败时回退到本地摘要器。
14. 支持用户编辑 `record.md`，`show` 默认读取可编辑版本。
15. 支持 Prompt 模板：`.expline/prompts/project_summary_prompt.txt` 和 `.expline/prompts/experiment_record_prompt.txt`。
16. 每次生成报告时会保存渲染后的 prompt 和 AI 原始响应。
17. 支持报告语言设置：`--report-language` 和 `expline config set report-language`。
18. 支持 `OPENAI_BASE_URL` 和 `expline config set openai-base-url`。
19. 支持 `--result-path` 记录轻量结果文件或目录摘要。
20. `--result-path` 只作为保存产物证据进入 Prompt，不用于解释指标变化原因或判断实验好坏。
21. 支持 Focused diff，优先把实验敏感代码和配置 diff 提供给 AI。
22. 支持 `diff-max-chars` 和 `focused-diff-max-chars` 配置。

## 仍有风险或偏差

1. `.expline/` 当前在工具仓库的 `.gitignore` 中被忽略，但需求文档希望用户实验项目中的 `.expline` 可以随 Git 同步。需要区分“ExpLine 工具仓库自身”与“用户实验项目”。
2. `expline config get/set` 会调用 `ensure_layout`，所以在未 init 的目录执行配置命令也会创建 `.expline/`，这与“先 init 再配置”的直觉略有偏差。
3. `run_user_command` 不捕获 stdout/stderr，只让实验命令直接输出。符合“不保存完整日志”的边界，但未来如果可视化页面需要运行摘要，目前缺少运行日志来源。
4. `diff.patch` 当前可能保存的是截断后的 diff，而不是完整原始 diff。严格按需求“保存本次 diff”理解，后续应区分完整 diff 和提供给 AI 的截断 diff。
5. `record.json` 字段与需求示例不完全一致，例如实现中使用 `evidence_index`，需求示例中写的是 `evidence_files`。
6. `rescan` 会覆盖 `project_summary.md`。如果用户手动编辑过项目语义说明，可能造成修改丢失。

## 建议下一步

优先实现 `expline rebuild`。

原因是 `rebuild` 是 Git 同步、多分支 merge、可视化页面和数据一致性检查的基础。建议先把事实源确定为每个实验目录中的 `record.json`，再由这些 JSON 重建：

1. 实验节点列表。
2. 父子边。
3. commit 到实验节点的索引。
4. 缺失 parent 和孤立节点报告。

在 `rebuild` 稳定后，再实现 `list` 和 `site`。静态 HTML 可视化应放在数据重建能力之后。
