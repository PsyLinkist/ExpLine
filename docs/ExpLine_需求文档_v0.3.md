# ExpLine 需求文档 v0.3

## 1. 项目定位

**ExpLine** 是一个面向科研代码实验的 **AI 自动实验语义记录与实验链路追踪工具**。

它的目标不是管理所有实验产物，也不是分析实验结果好坏，而是自动记录：

> 每次实验相比父实验，在代码、配置、命令和方法语义上发生了什么变化。

ExpLine 适用于科研人员、算法工程师、研究生等频繁修改代码并运行实验的人群，尤其适合“实验很多，但懒得手写实验记录”的场景。

---

## 2. 核心目标

ExpLine 需要解决以下问题：

1. 跑了很多实验后，忘记每次具体改了什么。
2. Git diff 真实但不够语义化，难以快速回忆实验含义。
3. 手写实验记录麻烦，容易漏记。
4. 实验之间存在继承、分叉、回溯修改，但缺少清晰链路。
5. 希望实验记录能随着项目 Git 仓库同步到云端。
6. 希望能用可视化方式查看整个项目的实验演化链。

ExpLine 的核心能力是：

```text
运行实验 → 捕获变化 → AI 生成语义记录 → 生成实验节点 → 更新实验链路 → 可视化展示
```

---

## 3. 项目边界

### 3.1 ExpLine 要做什么

ExpLine 需要做：

1. 初始化项目实验记录系统。
2. 扫描项目并生成项目语义说明。
3. 包装运行实验命令。
4. 每次实验运行后生成一个新的实验节点。
5. 自动捕获本次实验的代码、配置、命令变化。
6. 使用 AI 生成轻量语义实验记录。
7. 根据 Git commit 推断默认父实验。
8. 支持用户手动指定父实验。
9. 维护完整实验链路。
10. 生成静态可视化页面。
11. 支持实验记录随 Git 同步。

### 3.2 ExpLine 不做什么

MVP 阶段不做：

1. 不自动评价实验结果好坏。
2. 不解释指标上升或下降原因。
3. 不自动给出科研结论。
4. 不自动提出下一步实验方案。
5. 不自动修改用户代码。
6. 不自动调度多组实验。
7. 不保存完整源码快照。
8. 不保存大型实验输出。
9. 不强依赖 OpenClaw / Codex / Claude Code 等代码智能体。
10. 不优先做 VS Code 插件。
11. 不优先做复杂 Web 后端平台。

## 4. 基本使用方式

### 4.1 初始化项目

用户在项目根目录执行：

```bash
expline init
```

`init` 同时完成初始化和项目扫描，不要求用户额外执行 `scan`。

初始化后，ExpLine 需要：

1. 创建 `.expline` 工作目录。
2. 创建默认配置。
3. 创建默认 Prompt 模板。
4. 扫描当前项目结构。
5. 调用 AI 生成项目语义说明。
6. 初始化实验记录目录。
7. 初始化实验链路数据。
8. 生成初始静态可视化页面。

如果用户不希望调用 AI，可支持：

```bash
expline init --no-ai
```

如果项目结构后续发生较大变化，可支持：

```bash
expline rescan
```

### 4.2 运行并记录实验

用户原本这样运行实验：

```bash
python main.py --config config.yaml
```

改为：

```bash
expline run -- python main.py --config config.yaml
```

每次执行 `expline run` 时，ExpLine 需要：

1. 读取当前 Git HEAD commit。
2. 推断默认父实验。
3. 捕获当前代码和配置变化。
4. 运行用户实验命令。
5. 保存本次运行命令。
6. 保存本次 diff。
7. 保存变更文件列表。
8. 读取项目语义说明。
9. 读取父实验记录。
10. 调用 AI 生成本次实验语义记录。
11. 生成新的实验节点。
12. 更新实验链路。
13. 更新静态可视化页面。

### 4.3 手动指定父实验

当自动推断不符合用户意图时，用户可以显式指定父实验：

```bash
expline run --parent EXP-xxx -- python main.py
```

此时本次新实验节点的父实验固定为 `EXP-xxx`。

## 5. 实验节点更新规则

### 5.1 实验节点生成时机

实验节点不跟踪每次代码编辑，也不跟随每次 `git commit` 自动生成。

实验节点只在用户执行：

```bash
expline run -- <command>
```

时生成。

也就是说：

```text
一次 expline run = 一个新的实验节点
```

原因是实验节点应该对应一次实际实验运行，而不是一次普通代码修改。

### 5.2 节点只新增，不覆盖

每次实验生成一个新的实验节点。

旧实验节点不应被修改或覆盖。

如果用户从旧实验继续做新实验，应生成新的子节点，而不是改写旧节点。

### 5.3 节点内容应轻量

由于实验次数可能很多，每个实验节点只记录相对父实验的关键变化。

每个节点至少保存：

1. 实验 ID。
2. 父实验 ID。
3. 运行命令。
4. 当前 Git commit。
5. 当前 Git branch。
6. 本次 diff。
7. 变更文件列表。
8. AI 生成的一句话标题。
9. AI 生成的变化摘要。
10. 变更类型。
11. 受影响文件。
12. 相比父实验的语义差异。
13. 证据索引。

每个节点不默认保存：

1. 完整源码快照。
2. 完整项目扫描。
3. 完整环境信息。
4. 完整日志。
5. 大型实验输出。
6. 实验结果分析。

## 6. 默认父实验推断规则

### 6.1 核心原则

默认父实验不使用“最近一次实验”，也不使用全局 current 文件。

默认父实验应根据 **当前 Git HEAD commit** 推断。

核心规则：

> 当前实验的默认父实验 = 当前 Git HEAD commit 或其最近祖先 commit 上绑定的最近实验节点。

### 6.2 具体规则

当用户执行：

```bash
expline run -- <command>
```

且没有指定 `--parent` 时：

1. ExpLine 读取当前 Git HEAD commit。
2. 查找是否有实验节点绑定到当前 commit。
3. 如果当前 commit 上存在实验节点：
   - 若只有一个，使用该实验节点作为父实验。
   - 若有多个，使用该 commit 上最近创建的实验节点作为父实验。
4. 如果当前 commit 上没有实验节点：
   - 沿 Git 祖先 commit 向前查找最近一个绑定过实验节点的 commit。
   - 找到后，使用该 commit 上最近创建的实验节点作为父实验。
5. 如果没有找到任何对应实验：
   - 当前实验作为根实验。
6. 如果用户显式指定 `--parent`：
   - 始终使用用户指定的父实验，覆盖自动推断。

### 6.3 示例

假设 Git 历史为：

```text
C1  做实验 A
└── C2  做实验 B
```

实验记录为：

```text
C1 → EXP-A
C2 → EXP-B
```

用户后来回到 C1：

```bash
git checkout C1
git checkout -b new-idea
```

然后运行：

```bash
expline run -- python main.py
```

此时 ExpLine 发现当前 HEAD 是 C1，C1 对应 `EXP-A`，因此新实验默认接在 `EXP-A` 后面：

```text
EXP-A
├── EXP-B
└── EXP-C
```

这支持用户从旧实验状态回头开新分支。

### 6.4 同一 commit 多实验规则

同一个 Git commit 可能对应多个实验。

例如：

```text
C1
├── EXP-A: top_k=5
├── EXP-B: top_k=10
└── EXP-C: top_k=20
```

如果当前 HEAD 仍为 C1，且用户再次运行：

```bash
expline run -- python main.py --top_k 30
```

默认父实验为该 commit 上最近创建的实验，即 `EXP-C`。

如果用户希望从 `EXP-A` 继续，则手动指定：

```bash
expline run --parent EXP-A -- python main.py --top_k 30
```

## 7. Git commit 绑定需求

### 7.1 每个实验必须绑定 Git 状态

每个实验节点必须记录运行时的 Git 信息：

1. 当前 HEAD commit。
2. 当前 branch。
3. 工作区是否有未提交修改。
4. 本次 diff。
5. 变更文件列表。

Git commit 用于定位实验所在的代码历史位置。

diff 用于记录实际未提交修改。

### 7.2 Git commit 的作用

Git commit 不触发实验记录生成。

Git commit 只用于：

1. 推断默认父实验。
2. 定位实验对应的代码基础。
3. 支持实验分支关系推断。
4. 帮助用户回溯实验发生时的代码位置。

实验记录的触发仍然是：

```bash
expline run -- <command>
```

### 7.3 非 Git 仓库处理

如果当前项目不是 Git 仓库：

1. ExpLine 仍可运行。
2. 默认父实验无法通过 commit 推断。
3. 若无显式 `--parent`，当前实验作为根实验，或提示用户指定父实验。
4. 仍保存运行命令和变更摘要，但 Git diff 能力不可用。

## 8. AI 语义记录需求

### 8.1 AI 的职责

AI 只负责记录“实验变化”，不负责评判实验结果。

AI 需要生成：

1. 实验自动标题。
2. 一句话摘要。
3. 本次变更说明。
4. 变更类型标签。
5. 受影响文件。
6. 受影响模块或阶段。
7. 相比父实验的语义差异。
8. 原始证据索引。

AI 不生成：

1. 实验结果分析。
2. 指标变化原因。
3. 方法有效性判断。
4. 论文式结论。
5. 下一步实验建议。

### 8.2 AI 输入

每次生成实验记录时，AI 的输入包括：

1. 项目语义说明。
2. 父实验记录。
3. 当前运行命令。
4. 当前 Git diff。
5. 当前变更文件列表。
6. 必要的代码上下文片段。
7. 用户自定义 Prompt 模板。

### 8.3 AI 输出

每次实验需要生成两类输出。

#### Markdown 记录

用于人阅读，至少包括：

1. 实验 ID。
2. 父实验 ID。
3. 自动标题。
4. 运行命令。
5. 一句话摘要。
6. 本次变更说明。
7. 变更类型。
8. 受影响文件。
9. 相比父实验的语义差异。
10. 证据索引。

#### JSON 记录

用于可视化和链路重建，至少包括：

```json
{
  "experiment_id": "",
  "parent_id": "",
  "title": "",
  "summary": "",
  "command": "",
  "git_commit": "",
  "git_branch": "",
  "change_types": [],
  "affected_files": [],
  "affected_stages": [],
  "semantic_diff_from_parent": {
    "before": "",
    "after": ""
  },
  "evidence_files": [],
  "created_at": ""
}
```

### 8.4 自定义 Prompt 模板

ExpLine 必须支持用户自定义 Prompt 模板。

要求：

1. 初始化时生成默认 Prompt 模板。
2. 用户可以直接编辑模板。
3. 模板控制报告结构、语言、风格和关注重点。
4. 模板支持变量占位符，例如：
   - 项目语义说明
   - 父实验记录
   - 当前 diff
   - 当前命令
   - 当前变更文件列表
5. 每次生成记录时保存：
   - 使用的模板副本
   - 渲染后的完整 prompt
   - AI 原始输出

## 9. 项目初始化语义扫描需求

### 9.1 init 阶段项目扫描

`expline init` 需要扫描项目并生成项目语义说明。

项目语义说明用于给后续实验记录提供上下文。

需要包括：

1. 项目目标。
2. 主要入口文件。
3. 主要运行方式。
4. 核心代码模块。
5. 配置文件位置。
6. 实验相关脚本。
7. 输出目录。
8. 项目大致流程。
9. 可能影响实验的方法模块。
10. 可能影响实验的配置项。

### 9.2 项目语义说明不是每次重复生成

每个实验节点不重复保存完整项目说明。

项目语义说明只在：

1. `expline init`
2. `expline rescan`

时生成或更新。

每个实验节点只引用项目语义说明，并记录相对父实验的变化。

## 10. 实验链路需求

### 10.1 实验链路结构

ExpLine 需要维护整个项目的实验谱系。

每个节点是一次实验。

每条边表示父实验到子实验的继承关系。

支持：

```text
根实验
连续实验
分支实验
从旧实验回头继续修改
同一 commit 上多次实验
```

### 10.2 链路示例

```text
EXP-A
├── EXP-B
│   └── EXP-C
└── EXP-D
    └── EXP-E
```

其中：

1. `EXP-B` 和 `EXP-D` 都可以从 `EXP-A` 分叉。
2. `EXP-C` 继续继承 `EXP-B`。
3. `EXP-E` 继续继承 `EXP-D`。

### 10.3 链路重建

ExpLine 需要支持：

```bash
expline rebuild
```

`rebuild` 用于从所有实验 JSON 记录中重建：

1. 实验节点列表。
2. 父子边。
3. commit 到实验节点的索引。
4. 可视化页面数据。

`rebuild` 需要检查：

1. 重复实验 ID。
2. parent_id 不存在。
3. commit 索引缺失。
4. 孤立节点。

## 11. 静态可视化需求

### 11.1 可视化形式

MVP 使用静态 HTML 页面展示实验链路。

页面应展示：

1. 当前项目所有实验节点。
2. 父子有向边。
3. 分叉链条。
4. 实验标题。
5. 实验 ID。
6. 实验摘要。

### 11.2 页面交互

点击节点：

1. 显示该节点实验记录。
2. 显示标题、摘要、父实验、运行命令。
3. 显示变更类型。
4. 显示受影响文件。
5. 显示相对父实验的语义差异。
6. 显示证据路径。

右键节点：

MVP 至少支持：

1. 复制实验记录目录路径。
2. 复制实验报告路径。
3. 复制 diff 文件路径。
4. 复制 VS Code 打开命令。

纯静态 HTML 不强制支持直接打开文件夹。

未来可通过 `expline view` 轻量服务增强。

### 11.3 可视化更新时机

静态页面应在以下情况下更新：

1. `expline init` 后。
2. `expline run` 生成新实验后。
3. `expline rebuild` 后。
4. 用户手动执行 `expline site` 时。

## 12. 远程服务器可视化访问需求

### 12.1 使用场景

用户通常在大学实验室、远程 GPU 服务器或跳板机后的服务器上运行实验，并通过 VS Code Remote SSH 进行开发。

远程服务器通常没有图形界面，因此 ExpLine 的可视化页面不能依赖远程桌面环境，也不能要求用户在远程服务器上打开浏览器。

ExpLine 的可视化页面应支持：

```text
远程服务器生成实验记录与静态页面
↓
通过 VS Code Remote / SSH 端口转发
↓
本地浏览器查看和交互
```

---

### 12.2 推荐访问方式

ExpLine 的静态可视化页面应支持通过远程静态服务器 + VS Code 端口转发访问。

推荐流程：

```bash
expline site
ls -lh .expline/site/index.html
python -m http.server 8765 --bind 127.0.0.1 --directory .expline/site
```

然后用户在 VS Code 中：

```text
Ports 面板 → Forward Port → 8765
```

最后在本地浏览器访问：

```text
http://localhost:8765
```

该方式不要求远程服务器具有图形界面，也不需要暴露公网端口。

---

### 12.3 安全要求

远程可视化服务默认应只监听：

```text
127.0.0.1
```

不应默认监听：

```text
0.0.0.0
```

原因是 `0.0.0.0` 可能将实验记录页面暴露给服务器所在网络或公网。

如果未来提供：

```bash
expline view
```

则默认行为应为：

```bash
expline view --host 127.0.0.1 --port 8765
```

如果用户显式指定：

```bash
expline view --host 0.0.0.0
```

系统应给出安全提示。

---

### 12.4 可支持的三种查看方案

ExpLine 的静态可视化页面应兼容以下三种查看方式：

#### 方案一：VS Code Remote 直接预览

用户可以在 VS Code Remote 文件树中打开：

```text
.expline/site/index.html
```

并通过 VS Code 的 Preview / Live Preview 功能查看。

该方案使用简单，但复杂交互能力可能受 VS Code 预览环境限制。

#### 方案二：远程静态服务器 + VS Code 端口转发

用户在远程服务器启动本地静态服务：

```bash
expline site
ls -lh .expline/site/index.html
python -m http.server 8765 --bind 127.0.0.1 --directory .expline/site
```

然后通过 VS Code 端口转发在本地浏览器访问。

这是 MVP 推荐方案，最适合节点点击、右键菜单、搜索、高亮链路等交互。

#### 方案三：下载单文件 HTML 到本地打开

如果 `index.html` 为单文件 HTML，并内嵌实验链路与实验记录摘要，用户可以直接下载该文件到本地浏览器打开。

该方案适合离线查看，但对远程文件路径、VS Code 打开命令等功能支持较弱。

---

### 12.5 对静态页面生成的要求

为适配远程服务器场景，`expline site` 生成的页面应尽量采用单文件 HTML 设计：

1. 将实验链路数据内嵌到 `index.html`。
2. 将实验节点摘要内嵌到 `index.html`。
3. 尽量避免运行时依赖远程文件读取。
4. 点击节点时应能直接显示实验记录摘要或完整 Markdown 内容。
5. 即使用户将 `index.html` 下载到本地，也应能查看完整实验链路。
6. 与远程路径相关的功能应优先提供“复制路径”或“复制 VS Code 打开命令”。

---

### 12.6 节点交互需求

可视化页面需要支持以下节点交互能力。

#### 点击节点

点击某个实验节点后，页面应显示该实验详情，包括：

1. 实验 ID。
2. 实验标题。
3. 父实验 ID。
4. 运行命令。
5. 创建时间。
6. Git commit。
7. Git branch。
8. 一句话摘要。
9. 本次变更说明。
10. 变更类型。
11. 受影响文件。
12. 相比父实验的语义差异。
13. 证据文件列表。
14. 对应 `record.md` 内容。

#### 右键节点

右键某个实验节点后，页面应弹出菜单。

MVP 至少支持：

1. 复制实验 ID。
2. 复制实验记录目录路径。
3. 复制 `record.md` 路径。
4. 复制 `diff.patch` 路径。
5. 复制 `command.txt` 路径。
6. 复制 VS Code 打开实验目录命令。
7. 复制 VS Code 打开 `record.md` 命令。

纯静态 HTML 不要求直接打开远程文件夹。

未来如果提供 `expline view` 轻量服务，可增强为直接打开实验目录、报告文件或 diff 文件。

#### 高亮链路

点击节点后，页面应支持高亮：

1. 当前节点。
2. 父节点链路。
3. 子节点链路。

例如点击 `EXP-C` 后，高亮：

```text
EXP-A → EXP-B → EXP-C
```

#### 搜索节点

页面应提供搜索功能，支持按以下内容搜索：

1. 实验 ID。
2. 实验标题。
3. 摘要关键词。
4. 变更类型。
5. 受影响文件名。
6. Git commit。

搜索结果应高亮对应节点。

#### 查看证据内容

在节点详情区域中，应支持查看或展示以下证据内容：

1. `record.md`。
2. `diff.patch`。
3. `command.txt`。
4. `changed_files.txt`。

MVP 中可以优先展示 `record.md`，其他证据提供复制路径或折叠查看。

---

### 12.7 访问方式优先级

MVP 推荐优先支持：

```text
远程静态服务器 + VS Code 端口转发 + 本地浏览器查看
```

原因是该方案最接近标准浏览器环境，最适合复杂节点交互。

VS Code Preview 和本地下载 HTML 作为补充方案支持。


## 13. Git 同步需求

### 12.1 存储策略

ExpLine 采用项目内存储。

实验记录保存在当前项目的 `.expline` 目录中，并可随项目 Git 仓库一起同步。

目标是：

```text
项目代码在哪里，实验记录就在哪里。
```

### 12.2 需要同步的核心内容

建议纳入 Git 的内容：

1. 项目语义说明。
2. Prompt 模板。
3. 实验 Markdown 记录。
4. 实验 JSON 记录。
5. diff.patch。
6. command.txt。
7. changed_files。
8. lineage 数据。
9. commit 索引。
10. 可选静态页面。

### 12.3 可忽略内容

默认可忽略：

1. 缓存。
2. 临时文件。
3. 大型日志。
4. 大型实验输出。
5. 大型 artifacts。
6. 完整 stdout / stderr。
7. 可重新生成的站点文件，除非用户希望同步可视化页面。
8. AI 原始响应和完整 prompt，可由用户配置是否保存。

### 12.4 多分支 merge

用户可能在不同 Git 分支上产生不同实验记录。

这些实验记录通过 Git merge 合并。

容易冲突的文件包括：

1. lineage 数据。
2. commit 索引。
3. 静态页面。

ExpLine 需要支持通过：

```bash
expline rebuild
```

在 merge 后重建这些派生数据。

事实源应是每个实验节点的 JSON 记录。

## 14. 命令需求

### 13.1 MVP 核心命令

```bash
expline init
expline run -- <command>
expline list
expline show <experiment_id>
expline rebuild
expline site
```

### 13.2 可选命令

```bash
expline rescan
expline link --parent <parent_id> --child <child_id>
expline view
```

其中：

1. `rescan`：重新生成项目语义说明。
2. `link`：手动修正父子关系。
3. `view`：未来用于启动轻量服务，增强打开文件夹能力。

## 15. MVP 验收标准

第一版完成后，应满足：

1. `expline init` 能初始化项目并生成项目语义说明。
2. `expline run -- <command>` 能运行用户命令并生成实验节点。
3. 每个实验节点有唯一 ID。
4. 每个实验节点绑定当前 Git commit。
5. 默认父实验能根据当前 Git HEAD 或祖先 commit 推断。
6. 支持 `--parent` 手动指定父实验。
7. 每个实验节点只保存相对父实验的轻量变化记录。
8. AI 生成的是实验语义记录，而不是实验结果分析。
9. 每个实验同时生成 Markdown 和 JSON。
10. 支持用户自定义 Prompt 模板。
11. 支持分支实验链。
12. 支持从旧 commit 回头继续实验并自动接到正确父节点。
13. 支持 `rebuild` 重建实验链路和 commit 索引。
14. 支持静态 HTML 展示完整实验链。
15. 点击可视化节点能查看实验报告。
16. 右键节点至少能复制路径或 VS Code 打开命令。
17. 静态可视化页面支持远程服务器场景下通过 VS Code 端口转发在本地浏览器访问。
18. 静态页面的节点点击、右键复制、搜索和链路高亮应能在标准浏览器中正常工作。
19. `.expline` 可随项目 Git 仓库同步。
20. 多分支 merge 后可通过 `rebuild` 恢复完整实验图。

---

## 16. 一句话总结

**ExpLine 是一个 AI 驱动的轻量实验语义记录工具：用户通过 `expline run -- <command>` 运行实验，系统根据当前 Git commit 推断父实验，自动记录本次相对父实验的代码、配置、命令语义变化，并在静态可视化页面中展示整个项目的实验演化链。**
